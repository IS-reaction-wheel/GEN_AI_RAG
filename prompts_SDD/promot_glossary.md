## ユビキタス言語定義（glossary.md）の作成指示

必ずディレクトリ内の `CLAUDE.md`、`docs/product-requirements.md` (PRD)、および `docs/functional-design.md` (FD) を参照し、本プロジェクトのユビキタス言語定義（`docs/glossary.md`）を作成してください。

`CLAUDE.md` の規定に従い、開発者（AI アシスタント含む）とユーザー（製造業の現場エンジニア）の間で認識のズレを防ぎ、コード上の命名を統一するために、以下の項目を網羅してドキュメントを構成してください。

### 1. ドメイン用語・ビジネス用語の定義（製造業・業務コンテキスト）
PRD に記載されているターゲットユーザーの業務や対象データに関する用語を定義してください。
- 抽出対象の例: 不具合事例、品質管理、生産技術、製品設計、技術資料（ナレッジベース）、未加工データ

### 2. システム・AI 用語の定義（Agentic RAG コンテキスト）
FD およびアーキテクチャに記載されている、本システム特有の技術用語を定義してください。
- 抽出対象の例: Agentic RAG、自己修正ループ、タスク分割（Task Planning）、推論過程（Thinking/Reasoning）、チャンク（Chunk）、ハイブリッド検索、Reranking、ハルシネーション

### 3. UI/UX 用語の定義
Gradio を用いた UI 上でユーザーが触れる用語を定義してください。
- 抽出対象の例: システムプロンプト、Temperature、思考過程ログ、マルチターンチャット

### 4. 英語・日本語対応表（コード命名規則）
ドメイン設計とコードを一致させる（Domain-Driven Design のアプローチ）ため、上記の用語を Python コード（クラス名、変数名、辞書のキーなど）でどのように命名するか、厳密な対応表を作成してください。
以下の要素を必ず表に含めてください：
- **日本語名** (例: 質問, サブタスク, 目的, 要約, 判定, 回答, 不具合, 技術資料)
- **英語名 / 変数名** (例: `question`, `subtask`, `purpose`, `summary`, `judge` / `sufficient`, `answer`, `defect`, `document`)
- **適用箇所** (例: ワークフロー状態 `WorkflowState` のキー、Pydantic モデルのフィールド名など)

上記の条件を踏まえ、`docs/glossary.md` の内容をMarkdown形式で出力してください。
